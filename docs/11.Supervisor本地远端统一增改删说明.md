# Supervisor 本地远端统一增改删说明

## 背景与目标

此前仓库对 Supervisor 服务的写操作能力是不对称的：

- 新增只允许 `local` 主机
- 远端 `ansible` 主机只能做读取、运行操作和归档联动
- 没有正式的修改、删除 HTTP API

本次改造目标是把模板纳管服务的创建、修改、删除统一收敛到一套后端语义，并同时支持本地与远端主机。

## 当前问题或修改原因

- 前端只能新增，无法对已纳管服务做受控修改或硬删除。
- 远端主机虽然已经支持 `start/stop/restart` 和归档，但仍不能直接修改模板配置，能力边界不完整。
- 导入记录与模板记录共存在同一张主表里，如果缺少统一修改流程，后续纳管会越来越依赖人工处理。
- 删除如果不先停止服务，容易留下 Supervisor 仍持有进程、但配置文件和数据库已被删掉的现场分裂状态。

## 业务规则变更

- `POST /admin/api/supervisor/services` 放开到 `local + ansible` 两类主机。
- 新增 `PUT /admin/api/supervisor/services/{contentProgramName}?host=...`，用于修改未归档服务。
- 新增 `DELETE /admin/api/supervisor/services/{contentProgramName}?host=...`，用于硬删除已归档服务。
- 修改和删除同时适用于：
  - `TEMPLATE_MANAGED`
  - `IMPORTED_READONLY`
- `IMPORTED_READONLY` 记录一旦被修改，固定转成 `TEMPLATE_MANAGED`，并以新模板内容覆盖 `baseline_content`。
- `is_archived=true` 的记录禁止直接修改，但允许硬删除。
- `is_archived=false` 的记录不能直接删除，必须先归档。
- 删除固定流程：
  1. 先删除数据库记录
  2. 删除远端 `.bak`
  3. 若当前 `.ini` 仍存在，则一并删除
  4. 仅在实际删掉 `.ini` 时执行 `reread/update`
- 删除当前 `.ini/.bak` 已缺失时，不再强制失败；按“已是干净状态”处理。
- 硬删除会清理 `.bak`，不再保留归档备份。
- 修改时如果手动输入的 `contentProgramName`、`fileName` 或 `port` 导致 `contentProgramName/configPath` 变化：
  - 先停止旧服务
  - 备份旧 `.ini`
  - 写入新 `.ini`
  - 删除旧 `.ini`
  - 再 `reread/update`
- 若原记录来自 `/etc/supervisord.d` 子目录，修改后保留原父目录，只替换 basename。
- 端口冲突校验扩展为双重检查：
  - 递归扫描 `*.ini/.ini.bak/.ini.bak.*`
  - 执行 `ss -lnutp` 检查当前监听端口

## 接口或配置变更

新增和变更后的接口如下：

- `POST /admin/api/supervisor/services`
- `PUT /admin/api/supervisor/services/{contentProgramName}?host=...`
- `DELETE /admin/api/supervisor/services/{contentProgramName}?host=...`

接口约束：

- 创建请求使用 `host/jobName/moduleName/contentProgramName/javaPath/active/port/jarName/fileName/xms/xmx/user`
- 修改请求使用同一组业务字段，但 `host` 改为查询参数，不再放请求体
- 删除请求只使用路径中的旧 `contentProgramName` 和查询参数 `host`
- `PUT/DELETE` 成功响应都会返回 `commandResults`，便于前端展示本次远端执行结果

补偿规则：

- 创建成功写远端但写库失败时，立即删除刚写入的 `.ini` 并 `reread/update` 回滚
- 修改成功改完远端但写库失败时，优先从 `.bak` 恢复旧配置并 `reread/update`
- 硬删除以数据库删除结果为准；若删库成功但远端清理失败，接口返回成功并附带告警，不恢复主表记录

## 对调用方/运维的影响

- 前端后续不需要再区分“本地可写、远端不可写”的新增逻辑，新增、修改、删除都可以按统一接口接入。
- 前端发起删除前必须确保服务已经归档；未归档调用删除会直接返回 `409`。
- 若页面拿到 `409` 且文案为“服务未归档，需先归档后删除”，说明必须先归档，再执行硬删除。
- 对 `IMPORTED_READONLY` 记录执行修改后，记录会转成模板纳管语义；前端应按最新 `manageMode` 刷新展示。
- 运维排查端口冲突时，需要同时看数据库、配置目录和目标主机当前监听端口，不再只看 `.ini` 文件。

## 验证方式

- `python3 -m compileall app tests scripts`
- `.venv/bin/python -m pytest -q`
- 核心验证点：
  - 本地和远端都能创建服务
  - 本地和远端都能修改服务
  - 本地和远端都能硬删除已归档服务
  - 未归档记录不能直接删除
  - 归档记录不能被直接修改
  - 修改导入记录后会转成 `TEMPLATE_MANAGED`
  - 失败补偿后数据库与远端现场不发生分裂
