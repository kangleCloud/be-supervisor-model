# Supervisor 详情数据库化与单服务同步说明

## 背景与目标

之前的服务详情接口会在打开页面时直接读取远端 `.ini` 并执行 `supervisorctl status`。这导致详情接口职责过重，也让前端“打开详情”和“主动同步现场”这两个动作混在一起。

本次改造把 Supervisor 详情收敛为：

- 详情接口默认只查数据库
- 远端现场读取必须走独立同步接口
- 同步结果写回 `sys_supervisor_service`，后续详情继续读库

## 当前问题或修改原因

- 打开详情页就触发远端命令，页面行为不可预期，接口耗时也不稳定
- 详情接口同时承担“查库”和“拉现场”两类职责，难以向前端表达明确的交互语义
- 历史详情返回 `expectedContent/remoteContent/fileState`，不适合当前“库为主”的主数据模型

## 业务规则变更

- `GET /admin/api/supervisor/services/{programName}` 现在只返回数据库快照
- 详情接口不再隐式执行：
  - `supervisorctl status`
  - 远端 `.ini/.bak` 读取
  - 目录扫描
- 新增 `POST /admin/api/supervisor/services/{programName}/sync`
- 同步接口固定流程：
  1. 校验 JWT、主机白名单、服务存在性
  2. 已归档服务直接拒绝，同步错误码固定为 `40920`
  3. 按数据库中的 `programName` 读取 `supervisorctl status`
  4. 按数据库中的 `configPath` 读取当前 `.ini`
  5. 按需读取 `configPath + ".bak"`
  6. 解析 `command/directory/stdout_logfile` 与结构化字段
  7. 把结果写回 `sys_supervisor_service`
- 同步状态字段固定为：
  - `SUCCESS`：状态、配置读取和数据库更新都成功
  - `FAILED`：关键远端步骤缺失或失败，但数据库已记录本次失败快照
  - `UNKNOWN`：尚未执行过同步
- 主配置缺失时：
  - 本次同步记为 `FAILED`
  - `config_content` 清空
  - 已获取到的状态、备份存在性等字段仍尽量落库
- `.bak` 不存在属于正常状态，只通过 `has_backup=false` 表达，不单独报错

## 接口或配置变更

数据库新增字段：

- `command`
- `directory`
- `stdout_logfile`
- `has_backup`
- `config_content`
- `backup_config_content`
- `last_sync_at`
- `sync_status`
- `sync_error`

迁移规则：

- 详情同步相关字段由 Tortoise ORM 模型定义，并通过仓库根目录 `migrations/models/` 下的 Aerich migration 管理
- `app/database/migrations/001_init_schema.sql` 仅保留为历史参考快照，不再参与运行时初始化

新增接口：

```text
POST /admin/api/supervisor/services/{programName}/sync?host=...
```

详情响应核心字段：

- `host`、`hostName`
- `programName`、`configName`、`configPath`
- `jobName`、`moduleName`、`javaPath`、`active`、`port`、`jarName`、`xms`、`xmx`、`user`
- `command`、`directory`、`stdoutLogfile`
- `status`、`pid`、`uptime`
- `hasBackup`、`configContent`、`backupConfigContent`
- `lastSyncAt`、`syncStatus`、`syncError`
- `isArchived`、`archivedAt`、`restoredAt`

同步响应核心字段：

- `host`
- `programName`
- `status`、`pid`、`uptime`
- `syncedFields`
- `warnings`
- `lastSyncAt`
- `syncStatus`
- `syncError`
- `commandResults`

## 对调用方/运维的影响

- 前端打开详情页时，只调用 `GET detail`
- 前端需要最新现场数据时，必须额外调用 `POST sync`
- `POST sync` 成功后，前端应重新拉一次详情；如有需要，可再刷新列表
- 已归档服务仍允许查看详情，但不允许调用同步接口
- 运维如果需要排查同步失败，应优先看 `syncStatus/syncError` 和服务端日志，而不是再依赖旧版 `fileState`

## 验证方式

- `python3 -m compileall app tests scripts`
- `.venv/bin/python -m pytest -q`
- 手工验证建议：
  1. 先打开详情，确认接口不再触发远端读取
  2. 调用 `POST /admin/api/supervisor/services/{programName}/sync`
  3. 再次打开详情，确认 `status/configContent/lastSyncAt/syncStatus` 已更新
