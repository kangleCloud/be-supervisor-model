# 远端只读约束与路径语义修正

## 背景与目标

当前 Supervisor 列表接口已经切换为“数据库为主、远端为现场”的读取模型，但现场排查暴露出两个问题：

- 控制机加载 `SUPERVISOR_CONF_DIR=/etc/supervisord.d` 时，会被宿主机 `resolve()` 成 `/private/etc/supervisord.d`，导致远端 Linux 主机读取错目录
- `ansible` 读取目录时把 `find` 结果继续接 `| sort`，远端目录不存在时会被吞成空结果，误导为“查询成功但没有数据”

本次修改目标是修正路径语义、提升远端读取可靠性，并新增“远端主机只读”的系统硬约束。

## 当前问题或修改原因

- 现网远端主机已经存在 `/etc/supervisord.d/*.ini`，但数据库尚未导入主数据，列表接口自然返回空数组。
- 路径被错误改写成 `/private/etc/...` 后，即使执行导入脚本，也会因为读错远端目录而继续查不到配置。
- 远端目录不存在、权限异常或读取失败时，原实现可能表现成空列表或 `MISSING`，不利于快速定位问题。
- 当前项目已增加强约束：禁止通过本服务修改远端配置文件或触发远端 Supervisor 生效命令。

## 业务规则变更

- `SUPERVISOR_CONF_DIR` 和其他绝对路径配置只做词法归一化，不再做宿主机物理 `resolve()`。
- 路径边界校验继续限制在 `/etc/supervisord.d` 内，但返回路径保持 `/etc/...` 字面值，不再回写 `/private/...`。
- `executorType=ansible` 的主机只允许：
  - 读取配置文件
  - 查询 `supervisorctl status`
  - 执行导入脚本读取现网配置
- `executorType=ansible` 的主机固定禁止：
  - 写入、删除、备份、还原远端配置文件
  - 执行 `supervisorctl start/stop/restart/reread/update`
- `POST /admin/api/supervisor/services` 命中远端主机时直接返回 `403`，不再尝试写远端文件。
- 远端目录不存在、远端路径不可达或远端读取失败时，服务端必须返回明确错误，不能静默吞成空列表。

## 接口或配置变更

### 接口行为

- `POST /admin/api/supervisor/services`
  - `host` 对应 `local` 主机时，保持原有新增流程
  - `host` 对应 `ansible` 主机时，返回 `403`
  - 返回文案：`当前项目禁止修改远端配置文件`

### 配置语义

- `SUPERVISOR_CONF_DIR=/etc/supervisord.d`
  - 当前语义：读取配置后保持 `/etc/supervisord.d`
  - 不再因为控制机运行在 macOS 上被折叠成 `/private/etc/supervisord.d`

## 对调用方/运维的影响

- 调用方不能再通过新增接口改远端主机配置；远端主机目前只支持读。
- 现网远端主机切换到“库为主”模型前，仍需先执行导入脚本把已有 `.ini` 导入 `sys_supervisor_service`。
- 运维排查空列表时，应先区分：
  - 数据库未导入主数据
  - 远端读取目录错误
  - 远端目录不存在或读取失败

## 验证方式

- 运行 `python3 -m compileall app tests scripts/import_supervisor_services.py`
- 运行 `PYTHONPATH=. .venv/bin/pytest -q`
- 核心验证点：
  - `SUPERVISOR_CONF_DIR=/etc/supervisord.d` 时，配置值保持 `/etc/supervisord.d`
  - 路径校验仍能拦截穿越，但合法路径不再变成 `/private/...`
  - `ansible` 读取远端目录缺失时返回明确错误，不再静默返回空数组
  - 远端主机调用新增接口返回 `403`
  - 本地主机新增流程不受影响
