# 数据库基线DDL整合说明

## 背景与目标

当前仓库原先通过 `001 + 002 + 003 + 004` 多个迁移文件逐步拼出最终库结构，阅读和初始化成本都偏高。

本次调整的目标是把当前最终数据库结构整理成一份单基线初始化 SQL，保证新环境初始化时只需要执行一个文件就能得到完整表结构和默认管理员数据。

## 当前问题或修改原因

- 认证表、Supervisor 主数据表和默认管理员初始化分散在多份 SQL 中，不便于统一审阅 DDL。
- `sys_supervisor_service` 的最终字段依赖 `003` 建表后再由 `004` 扩展，阅读时必须手工合并结构。
- 当前环境已经允许重置迁移历史，不再需要维护旧的原地升级链路。

## 业务规则变更

- `app/database/migrations` 目录现在只保留 `001_init_schema.sql`。
- `001_init_schema.sql` 同时包含：
  - `sys_user`
  - `sys_login_log`
  - `sys_login_token`
  - `sys_supervisor_service`
  - 默认超级管理员 `admin` 初始化 SQL
- `sys_supervisor_service` 的最终结构直接以当前代码真实依赖的字段为准，不再保留“先建旧结构再增量扩展”的历史表达。
- `sys_schema_migration` 版本记录重置为单版本，初始化完成后只记录版本 `1`。

## 接口或配置变更

- `app/core/database.py` 的缺表补建逻辑改为重放单基线 SQL。
- `sys_supervisor_service` 的最终唯一键固定为 `(host_ip, config_path)`。
- 文档中的数据库初始化说明统一改成“执行单基线初始化 SQL”，不再描述多文件增量迁移链。

## 对调用方/运维的影响

- 新环境初始化时，运维只需要关注 `app/database/migrations/001_init_schema.sql`。
- 已清空数据表的环境不再需要保留旧迁移版本记录，也不需要按旧顺序理解 `001-004` 的演进关系。
- 默认管理员初始化逻辑仍保留幂等保护，多次初始化不会重复插入 `admin`。

## 验证方式

- 运行 `python3 -m compileall app tests scripts`
- 运行 `.venv/bin/python -m pytest -q`
- 验收重点：
  - `sys_schema_migration` 只记录版本 `1`
  - 缺失 `sys_login_token` 或 `sys_supervisor_service` 时，单基线仍能自动补建
  - `001_init_schema.sql` 中同时存在认证表、Supervisor 主数据表最终字段、最终索引和默认管理员初始化 SQL
