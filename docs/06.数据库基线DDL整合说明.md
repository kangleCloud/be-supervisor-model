# 数据库基线DDL整合说明

## 背景与目标

当前仓库的运行时数据库链路已经切换到 Tortoise ORM + Aerich，应用启动阶段不再负责自动建库、补表或执行历史 SQL。

本次调整的目标是把数据库 source of truth 收敛到：

- `app/database/models/` 下的 Tortoise 模型
- 仓库根目录 `migrations/models/` 下的 Aerich migration

同时保留 `app/database/migrations/001_init_schema.sql` 作为历史 DDL 参考快照，便于人工对照和追溯。

## 当前问题或修改原因

- 旧的 `app/core/database.py` 仍然保留了自动建库、自动补表和 `sys_schema_migration` 逻辑，和当前 ORM 链路重复。
- `001_init_schema.sql` 过去同时承担“历史快照”和“运行时初始化入口”两种职责，容易误导后续改造。
- 仓库此前没有真实提交 Aerich baseline 目录，导致模型与迁移之间缺少正式落库成果。

## 业务规则变更

- 删除 `app/core/database.py`，应用运行时不再保留旧的同步建库/补表入口。
- 应用启动只执行 `init_database()` / `close_database()`，不再自动创建数据库、执行 SQL migration 或补种默认管理员。
- schema 变更统一通过 Aerich 管理，仓库根目录 `migrations/models/` 为唯一迁移来源。
- 新增真实 Aerich baseline migration，覆盖：
  - `sys_user`
  - `sys_login_log`
  - `sys_login_token`
  - `sys_supervisor_service`
  - `sys_supervisor_import_staging`
- 默认超级管理员 `admin` 初始化 SQL 随 baseline 一并落库，不再依赖应用启动逻辑。
- `app/database/migrations/001_init_schema.sql` 仅保留为历史 DDL 参考快照，不再承担任何运行时职责。

## 接口或配置变更

- `pyproject.toml` 中 `[tool.aerich]` 继续指向 `app.database.bootstrap.TORTOISE_ORM`，migration 目录固定为仓库根 `./migrations`。
- 运维初始化数据库的标准流程改为：先由外部环境创建数据库，再执行 `aerich upgrade`。
- `sys_supervisor_service`、`sys_supervisor_import_staging` 等表结构的新增/修改，统一先改模型，再生成 Aerich migration。

## 对调用方/运维的影响

- 数据库本身必须由外部环境预先创建，应用启动不会自动 `CREATE DATABASE`。
- 新环境初始化时应关注 `migrations/models/` 下的 Aerich baseline，而不是依赖 `001_init_schema.sql`。
- 若需要默认管理员，执行 `aerich upgrade` 即可落库，无需等待应用首次启动。
- `001_init_schema.sql` 仍可用于人工审阅历史表结构，但不能再作为运行命令或补表依据。

## 验证方式

- 运行 `python3 -m compileall app tests scripts`
- 运行 `.venv/bin/python -m pytest -q`
- 验收重点：
  - `migrations/models/` 存在真实 Aerich baseline 文件和 `__init__.py`
  - baseline 中包含 5 张业务表与默认 `admin` 初始化 SQL
  - `rg "core\\.database|initialize_database\\(|connect_database\\(|get_connection\\(|sys_schema_migration" app tests docs README.md` 不再命中运行时链路
  - 应用启动只初始化/关闭 ORM 连接，不再尝试自动建库或自动迁移
