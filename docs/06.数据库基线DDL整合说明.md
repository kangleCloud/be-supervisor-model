# 数据库基线DDL整合说明

## 背景与目标

当前仓库原先通过 `001 + 002 + 003 + 004` 多个迁移文件逐步拼出最终库结构，阅读和初始化成本都偏高。

本次调整的目标是把当前最终数据库结构整理成一份完整基线初始化 SQL，保证新环境初始化时主要依赖一个文件就能得到完整表结构和默认管理员数据；同时对少量旧库保留最小兼容升级迁移。

## 当前问题或修改原因

- 认证表、Supervisor 主数据表和默认管理员初始化分散在多份 SQL 中，不便于统一审阅 DDL。
- `sys_supervisor_service` 的最终字段依赖 `003` 建表后再由 `004` 扩展，阅读时必须手工合并结构。
- 当前环境以新库初始化为主，但仍存在少量历史库可能只应用过旧版 `001`，需要补齐 `sys_supervisor_service` 运行时字段。

## 业务规则变更

- `app/database/migrations/001_init_schema.sql` 同时包含：
  - `sys_user`
  - `sys_login_log`
  - `sys_login_token`
  - `sys_supervisor_service`
  - 默认超级管理员 `admin` 初始化 SQL
- `app/database/migrations/002_add_supervisor_service_runtime_columns.sql` 保留为历史库兼容迁移：
  - 只负责给旧版 `sys_supervisor_service` 补 `status/pid/uptime/status_sync_time`
  - 以及补 `idx_supervisor_host_status`
  - 新库初始化时若 001 已具备这些列，002 会被安全跳过
- `sys_supervisor_service` 的最终结构直接以当前代码真实依赖的字段为准，不再保留“先建旧结构再增量扩展”的历史表达。
- `sys_schema_migration` 初始化完成后会记录版本 `1` 和 `2`，但新库的最终结构仍以 `001` 为准。

## 接口或配置变更

- `app/core/database.py` 的缺表补建逻辑继续重放 `001_init_schema.sql`。
- `002` 的执行逻辑改为先检查列和索引是否存在，再按缺失状态补建。
- `sys_supervisor_service` 的最终唯一键固定为 `(host_ip, config_path)`。
- 文档中的数据库初始化说明统一改成“新库走完整基线，旧库按需执行兼容补齐迁移”。

## 对调用方/运维的影响

- 新环境初始化时，运维主要关注 `app/database/migrations/001_init_schema.sql`。
- 旧环境若历史上只应用过旧版 001，也不需要手工改 SQL；服务启动时会自动判断是否需要执行 002。
- 默认管理员初始化逻辑仍保留幂等保护，多次初始化不会重复插入 `admin`。

## 验证方式

- 运行 `python3 -m compileall app tests scripts`
- 运行 `.venv/bin/python -m pytest -q`
- 验收重点：
  - 空库初始化后 `sys_schema_migration` 记录版本 `1` 和 `2`
  - 缺失 `sys_login_token` 或 `sys_supervisor_service` 时，001 仍能自动补建
  - 旧版 `sys_supervisor_service` 缺少运行时字段时，002 能安全补齐
  - `001_init_schema.sql` 中同时存在认证表、Supervisor 主数据表最终字段、最终索引和默认管理员初始化 SQL
