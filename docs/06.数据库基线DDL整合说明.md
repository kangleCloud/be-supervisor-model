# 数据库基线 DDL 与手工升级说明

## 背景与目标

数据库结构继续由 Tortoise ORM 模型约束，但不再依赖额外迁移工具；后续 schema 统一通过仓库内 SQL 文件手工执行。

## 当前问题或修改原因

自动迁移链路会把部署可用性绑定到额外工具、额外格式和额外元数据；只要工具链不兼容，应用会在业务代码启动前直接退出。

## 业务规则变更

- `app/database/migrations/001_init_schema.sql` 改为新库初始化基线 SQL
- `app/database/migrations/002_fix_supervisor_service_legacy_schema.sql` 改为旧库升级 SQL
- 仓库不再保留 Python 迁移文件
- 应用启动只初始化数据库连接，不自动迁移、不自动修库
- MySQL 启动时继续做 schema fail-fast：
  - 必须存在 `config_path/file_name/content_program_name`
  - 不允许残留旧版 `program_name/config_name`
  - 不允许残留旧唯一键和旧索引

## 接口或配置变更

- 无新增 HTTP 接口
- 运维 SQL 规则固定为：
  - 新库先执行 `app/database/migrations/001_init_schema.sql`
  - 旧库升级时执行 `app/database/migrations/002_fix_supervisor_service_legacy_schema.sql`
- 启动失败文案固定直接指向对应 SQL 文件

## 对调用方/运维的影响

- 新库初始化和旧库升级都改为手工执行 SQL
- 若启动提示 schema 落后，应先执行仓库内 SQL，再拉起应用
- 是否额外清理历史迁移元数据，由运维自行决定，不影响当前应用启动

## 验证方式

- 新库执行 `001_init_schema.sql` 后可正常启动
- 旧库执行 `002_fix_supervisor_service_legacy_schema.sql` 后可正常启动
- 旧库残留旧字段时启动会直接失败，并给出明确 SQL 路径
