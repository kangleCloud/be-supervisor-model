# Supervisor服务器概况真实API说明

## 背景与目标

当前前端在主机切换、页面刷新和手动点击“刷新服务器概况”时，需要拿到目标主机的实时 CPU、内存和基础可用性信息。

本次收口保持概况接口为唯一真实数据来源，由后端即时采集现场数据后直接返回给前端，不落库、不起线程、不做后台轮询，也不允许页面回退到 mock 数据。

## 当前问题或修改原因

- 概况接口已经存在，但历史实现把采集超时写死为 `8s`，与现有 ansible 读取链路配置割裂。
- 服务器概况属于瞬时现场信息，不适合写入 `sys_supervisor_service` 或引入额外任务表。
- 前端需要一个稳定的结构化结果，来区分“已连接可用”“暂时不可达”“当前版本不支持采集”三类状态，同时禁止自行拼装 mock 值。

## 业务规则变更

- 新增 `GET /admin/api/supervisor/overview?host=...`。
- 目标主机仍然必须通过现有白名单校验和 JWT 鉴权。
- 接口只执行一次固定受控的只读脚本，不允许透传任意命令。
- v1 仅保证远端 `ansible` Linux 主机真实可用。
- 概况采集不再保留 `8s` 特例，统一复用 `executor.timeoutSeconds` / `ANSIBLE_COMMAND_TIMEOUT_SECONDS`。
- 远端 ansible 调用前先在控制机校验 `executor.inventoryPath` 是否存在且可读；本地配置错误必须直接返回可排查提示。
- `local` 主机固定返回：
  - `available=false`
  - `connectionState=UNSUPPORTED`
- 远端主机暂时不可达时固定返回 `200`，但：
  - `available=false`
  - `connectionState=UNREACHABLE`
- inventory 缺失、inventory 不可读、SSH 超时、UNREACHABLE 等远端失败场景都保持 `200 + UNREACHABLE`，但 `warnings` 必须保留真实根因摘要。
- 不可达或不支持时，数值字段继续返回 `0 / 0.00% / 0 B`；调用方必须用 `connectionState + warnings + checks` 判断状态，不能把 0 值当成真实主机负载。
- 响应头固定：`Cache-Control: no-store`
- 不写数据库、不启线程、不增加定时器、不生成历史曲线。

## 接口或配置变更

### 请求

```text
GET /admin/api/supervisor/overview?host=10.1.0.104
```

请求参数：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| host | string | 是 | 目标主机 IP，继续沿用现有白名单校验 |

### 响应字段

`data` 固定包含：

- `host`
- `hostName`
- `executorType`
- `available`
- `connectionState`：`CONNECTED | UNREACHABLE | UNSUPPORTED`
- `collectedAt`
- `cpu.usagePercent`
- `memory.usagePercent`
- `memory.usedBytes`
- `memory.totalBytes`
- `memory.usedText`
- `memory.totalText`
- `checks.supervisorctlAvailable`
- `checks.confDirReadable`
- `warnings`

## 前端对接规则

- Supervisor 页面所有展示数据都必须走后端接口，不允许前端回退到 mock CPU、mock 内存、mock 检查项或 mock 状态。
- 主机下拉来源固定为 `GET /admin/api/supervisor/hosts`。
- 概况卡片来源固定为 `GET /admin/api/supervisor/overview`。
- 服务列表来源固定为 `GET /admin/api/supervisor/services`。
- 状态刷新来源固定为 `POST /admin/api/supervisor/services/status/refresh`。
- 服务详情来源固定为 `GET /admin/api/supervisor/services/{contentProgramName}`。
- 页面刷新、切换主机、点击“刷新服务器概况”时，都必须重新请求概况接口。
- 前端若做页面内短暂展示缓存，也只能缓存最近一次真实接口响应；不得脱离接口结果自行构造数据。
- 概况卡片展示必须基于 `connectionState + checks + warnings + hostName` 渲染，不得把 `cpu=0`、`memory=0` 直接解释成主机空闲。

## 对调用方/运维的影响

- 服务器概况不再依赖数据库，权威数据来自实时采集结果。
- 前端需要接受 `200 + available=false` 的合法返回，而不是把它当作接口失败。
- 概况接口、导入接口、同步接口都会复用同一套 ansible 基础配置：`inventoryPath`、`remoteUser`、`timeoutSeconds`。
- inventory 中必须能直接按目标主机 IP 匹配，控制机必须能使用 `remoteUser` 免交互访问目标主机。
- 运维排查问题时，应优先看 `connectionState` 与 `warnings`，而不是假设所有失败都走 HTTP 5xx。
- v1 不覆盖本机 macOS / 非 Linux 采集场景，相关环境统一返回 `UNSUPPORTED`。

## 验证方式

- 运行 `python3 -m compileall app tests scripts`
- 运行 `.venv/bin/python -m pytest -q`
- 重点验收：
  - 白名单远端主机返回完整结构
  - 远端不可达时返回 `200 + available=false + connectionState=UNREACHABLE`
  - `local` 主机返回 `200 + available=false + connectionState=UNSUPPORTED`
  - 接口响应头包含 `Cache-Control: no-store`
  - 不写数据库，不影响现有导入、增改删、同步、运行、归档链路
