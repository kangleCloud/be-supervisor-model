# Supervisor初始化导入暂存恢复说明

## 背景与目标

初始化导入的 `PRECHECK` 在远端文件较多时耗时较长。前端如果在预检后刷新页面、切换主机或重新打开弹窗，之前的预检结果无法恢复展示，导致用户只能重复扫描远端。

本次目标是补齐“按主机恢复最近预检结果”的接口能力，让前端优先读取暂存结果，再决定是否重新预检或直接提交导入。

## 当前问题或修改原因

- 现有 `PRECHECK` 虽然已经把结果写入 `sys_supervisor_import_staging`，但缺少对外读取入口。
- 前端只能依赖内存中的 `batchId`，页面刷新后上下文丢失。
- 用户重进弹窗后看不到上一次预检结果，也无法直接继续 `COMMIT`。

## 业务规则变更

- 保留 `POST /admin/api/supervisor/imports` 的两阶段协议：`PRECHECK -> COMMIT`。
- 新增 `GET /admin/api/supervisor/imports/staging?host=...`：
  - 按“当前登录用户 + 主机 IP”读取最近一批暂存结果
  - 无数据时返回空状态
  - 有数据时返回 `batchId + createdAt + summary + items`
- `PRECHECK` 仍然固定先清理“该用户 + 该主机”的旧暂存，再重新扫描远端并写入新批次。
- `COMMIT` 仍然固定整批事务提交：
  - 全部成功才写入正式表
  - 成功后删除该批次暂存
  - 任一条失败则正式表整批回滚，暂存保留
- 暂存恢复不做跨用户共享，不支持同用户同主机保留多历史批次。

## 接口或配置变更

新增接口：

```text
GET /admin/api/supervisor/imports/staging?host=10.1.0.104
```

响应 `data` 固定包含：

- `host`
- `exists`
- `batchId`
- `createdAt`
- `summary`
- `items`

响应语义：

- `exists=false`：当前用户在该主机无可恢复暂存
- `exists=true`：返回最近一批预检结果，前端可直接展示

前端约定：

- 打开导入弹窗、切换主机、页面刷新时先调 `GET staging`
- 只有 `planned > 0` 且 `skipped = 0` 时才显示或启用“导入”按钮
- 点击“预检”时始终发起新的 `PRECHECK`
- 点击“导入”时使用 `GET staging` 或最近一次 `PRECHECK` 返回的 `batchId` 发起 `COMMIT`

## 对调用方/运维的影响

- 前端不再需要单纯依赖本地缓存 `batchId` 才能继续导入。
- 用户在预检后刷新页面或重新进入弹窗，仍可继续查看并提交最近一批结果。
- 运维脚本逻辑不变，仍复用现有 `PRECHECK -> COMMIT` 服务层。

## 验证方式

- 运行 `python3 -m compileall app tests scripts`
- 运行 `.venv/bin/python -m pytest -q`
- 核心验收：
  - `PRECHECK` 后 `GET staging` 能恢复最近批次
  - 同一用户重复 `PRECHECK` 会覆盖旧暂存
  - 不同用户互相看不到对方暂存
  - `COMMIT` 成功后暂存删除
  - `COMMIT` 失败时暂存保留且正式表不出现部分写入
