# Docker 部署与日志输出说明

## 背景与目标

仓库需要提供可直接落地的容器部署方案，同时允许在 `docker-compose.yml` 中显式定义宿主机日志输出路径。

## 当前问题或修改原因

如果容器启动链路依赖额外迁移命令，部署可用性会被数据库工具链耦合；而日志侧仍需要同时兼顾 `docker logs` 与宿主机历史留存。

## 业务规则变更

- 应用日志固定采用 `stdout + 文件双写`
- 未配置 `APP_LOG_PATH` 时仅 stdout
- 配置 `APP_LOG_PATH` 后启用滚动文件日志
- 日志滚动策略固定：
  - `maxBytes=50MB`
  - `backupCount=10`
- 容器启动前不再自动执行任何迁移命令

## 接口或配置变更

### Dockerfile

- 基于 `python:3.12-slim`
- 安装：
  - Python 依赖
  - `ansible-core`
  - `openssh-client`
  - `bash`
- 默认启动命令：`bash ./scripts/run.sh prod`

### docker-compose.yml

- 只部署应用容器
- 继续连接外部 MySQL
- 挂载：
  - `./config.yaml:/app/config.yaml:ro`
  - `/etc/ansible:/etc/ansible:ro`
  - `/data/logs/be-supervisor-model/:/var/log/be-supervisor-model`
- 环境变量：
  - `APP_ENV=prod`
  - `APP_CONFIG_PATH=/app/config.yaml`
  - `APP_LOG_PATH=/var/log/be-supervisor-model/app.log`
- 启动前置动作改为运维手工执行：
  - 新库：先执行 `app/database/migrations/001_init_schema.sql`
  - 旧库：按需执行 `app/database/migrations/002_fix_supervisor_service_legacy_schema.sql`
- compose 启动时只执行应用启动脚本，不再预执行数据库迁移

## 对调用方/运维的影响

- 宿主机必须提前准备 `/etc/ansible`
- 宿主机必须提前准备 `/data/logs/be-supervisor-model/`
- 运维必须在拉起容器前手工执行仓库内 SQL 文件
- 运维既可以通过 `docker logs` 看实时输出，也可以从 `/data/logs/be-supervisor-model/` 查看历史文件日志

## 验证方式

- `docker compose config`
- 先手工执行 SQL，再 `docker compose up -d --build`
- 检查容器内外日志路径是否同时有输出
- 检查 compose 启动链路不再执行数据库迁移命令
