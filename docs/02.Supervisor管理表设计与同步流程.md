# Supervisor管理表设计与同步流程

## 背景与目标

原实现的服务列表和详情长期依赖远端 `/etc/supervisord.d` 全量扫描，带来两个问题：

- 页面主视图和远端文件强耦合，无法明确区分“期望配置”和“现场状态”
- 现网存在人工改文件、文件丢失或未导入配置时，接口很难稳定表达问题

本次改造的目标是引入 `sys_supervisor_service` 作为 Supervisor 配置主数据表，把服务读取流程切换为“数据库为主、远端文件为执行结果”。

## 当前问题或修改原因

- 原接口同时承载读取、修改、删除、备份、还原和进程操作，职责过重，不适合作为 v1 主数据能力。
- 原列表接口按远端目录扫描结果返回，数据库中没有服务主数据，无法支撑后续配置治理。
- 远端目录里可能存在库外人工文件，必须有数据库唯一性和远端扫描两道校验。

## 业务规则变更

- v1 Supervisor 对外接口只保留：
  - `GET /admin/api/supervisor/hosts`
  - `GET /admin/api/supervisor/services`
  - `GET /admin/api/supervisor/services/{programName}`
  - `POST /admin/api/supervisor/services`
- 新增服务时，先完成远端 `.ini` 写入和 `reread/update`，成功后再写数据库。
- 列表和详情固定先查 `sys_supervisor_service`，不再以远端扫目录作为主视图。
- 详情接口新增 `expectedContent` 与 `fileState`：
  - `MATCH`：远端文件与数据库渲染结果一致
  - `MISSING`：数据库有记录，但远端文件不存在
  - `MISMATCH`：远端文件存在，但内容与数据库渲染结果不一致
- 如果远端写成功但数据库写失败，服务端必须立即删除刚写入的 `.ini`，再执行一次 `reread/update` 回滚。

## 接口或配置变更

### 数据表

当前 `sys_supervisor_service` 已并入单基线初始化 SQL，最终结构直接包含：

- 现场定位字段：`config_path`、`file_name`、`content_program_name`
- 纳管模式字段：`manage_mode`、`baseline_content`、`metadata_complete`、`parse_warnings`
- 模板结构化字段：`job_name`、`module_name`、`program_name`、`config_name`、`java_path`、`active_profile`、`port`、`jar_name`、`xms`、`xmx`、`run_user`
- 真实唯一约束：`(host_ip, config_path)`

### 请求模型

`POST /admin/api/supervisor/services` 只接受以下字段：

- `host`
- `jobName`
- `moduleName`
- `javaPath`
- `active`
- `port`
- `jarName`
- `configName`
- `xms`
- `xmx`
- `user`

`autoStart` 在 v1 中已下线，不再接受。

### 一次性导入脚本

新增 `scripts/import_supervisor_services.py`，用于把现网已有 `*.ini` 导入 `sys_supervisor_service`。

导入限制：

- 只导入 `*.ini`
- 不导入 `.bak` 和归档备份
- 只能导入可完整反解字段的配置

## 对调用方/运维的影响

- 调用方需要按新接口范围接入，不能再依赖修改、删除、备份、还原和独立状态接口。
- 现网切换前，运维必须先执行一次导入脚本，否则旧的远端 `.ini` 不会出现在列表和详情接口中。
- 当页面看到 `fileState=MISSING` 或 `MISMATCH` 时，表示数据库主数据和远端现场已不一致，需要先修复现场。

## 验证方式

- 运行 `python3 -m compileall app scripts/import_supervisor_services.py`
- 运行 `.venv/bin/python -m pytest -q`
- 核心验证点：
  - 创建成功后远端生成 `.ini` 且数据库新增记录
  - 列表来源于数据库，而不是远端扫目录
  - 远端文件缺失时返回 `MISSING`
  - 远端文件被手工改动时返回 `MISMATCH`
  - 数据库写入失败时，远端新配置会被自动回滚
