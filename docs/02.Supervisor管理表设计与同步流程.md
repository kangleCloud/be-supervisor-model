# Supervisor 管理表设计与同步流程

## 背景与目标

本专题用于明确 `sys_supervisor_service` 的当前职责、字段分工，以及“数据库为主、远端显式同步”的运行模式。

## 当前问题或修改原因

历史实现长期混用“扫远端目录”和“读数据库”，会导致列表不稳定、详情打开即触发远端读取、导入记录和模板记录语义不一致。

## 业务规则变更

- `sys_supervisor_service` 是 Supervisor 服务主数据唯一来源
- 列表和详情默认只查数据库
- 远端同步必须显式调用同步接口
- 主表真实定位字段统一为：
  - `config_path`
  - `file_name`
  - `content_program_name`
- 主表保留结构化字段、运行快照字段、归档字段和导入快照字段

字段分工：

- `baseline_content`：模板基线或导入时保存的原始快照
- `config_content`：最近一次显式同步到的当前 `.ini`
- `backup_config_content`：最近一次显式同步到的 `.bak`
- `manage_mode`：`TEMPLATE_MANAGED` / `IMPORTED_READONLY`
- `sync_status`：最近一次详情同步状态

## 接口或配置变更

- `GET /admin/api/supervisor/services`
  - 纯数据库分页
- `GET /admin/api/supervisor/services/{contentProgramName}`
  - 纯数据库详情
- `POST /admin/api/supervisor/services/{contentProgramName}/sync`
  - 读取远端状态与配置并回写数据库

## 对调用方/运维的影响

- 前端不应再依赖“打开详情自动读远端”
- 现网配置接入前，必须先完成初始化导入
- 运维定位配置文件应始终以 `configPath` 为准，而不是仅看文件名或显示名称

## 验证方式

- 列表接口不触发远端命令
- 详情接口不读取远端 `.ini/.bak`
- 同步接口能更新 `status/pid/uptime/configContent/backupConfigContent`
