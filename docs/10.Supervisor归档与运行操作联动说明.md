# Supervisor 归档与运行操作联动说明

## 背景与目标

归档后的服务必须与运行操作、还原和硬删除形成一致的状态机，避免接口语义互相冲突。

## 当前问题或修改原因

如果归档记录仍允许启动或直接修改，会破坏“数据库状态”和“远端配置现场”之间的一致性。

## 业务规则变更

- 未归档记录允许：
  - `start/stop/restart`
  - 修改
  - 归档
- 已归档记录禁止：
  - `start/stop/restart`
  - 修改
- 已归档记录允许：
  - 查看详情
  - 还原
  - 硬删除

归档流程固定为：

1. `stop`
2. 备份 `.ini -> .bak`
3. 删除 `.ini`
4. `reread/update`
5. 标记 `is_archived=true`

还原流程固定为：

1. 恢复 `.bak`
2. `reread/update`
3. 同步状态快照
4. 不自动启动

## 接口或配置变更

- `POST /services/{contentProgramName}/archive`
- `POST /services/{contentProgramName}/restore`
- `POST /services/{contentProgramName}/start`
- `POST /services/{contentProgramName}/stop`
- `POST /services/{contentProgramName}/restart`

## 对调用方/运维的影响

- 前端应根据归档状态控制按钮展示，但后端会继续做硬校验
- 归档后如需彻底移除，必须走删除接口，而不是直接删数据库

## 验证方式

- 归档后运行操作被拒绝
- 还原后可再次运行
- 归档时远端 `.ini` 被移除、`.bak` 被保留
