# 只读初始化导入与冗余字段设计

## 背景与目标

当前 Supervisor 列表接口已经固定为“数据库为主、远端为现场”的读取模型，因此远端主机在切换前必须先把现场 `.ini` 初始化导入 `sys_supervisor_service`。

针对 `10.1.0.104` 的排查已经确认：

- 远端主机可只读连通
- `/etc/supervisord.d` 顶层存在 11 个活跃 `.ini`
- 递归子目录后共有 59 个活跃 `.ini`
- 当前数据库中 `host_ip=10.1.0.104` 的记录数为 `0`

本次改造目标是让系统支持“只读扫描远端 + 写数据库快照”，同时显式保存文件名与配置内容中的 `program_name`，避免 legacy 配置被当前模板语义强行覆盖。

## 当前问题或修改原因

- 旧实现只扫描 `/etc/supervisord.d` 顶层，天然漏掉子目录中的大量活跃配置。
- 旧导入脚本要求完整反解模板字段，legacy 文件一旦存在重复 key、目录结构不标准或字段缺失，就会整文件跳过。
- 旧模型把 `config_name/program_name` 同时当成展示字段和真实定位键，无法表达“文件名与内容里的 programName 不一致但都要保留”的现场事实。
- 远端主机当前明确禁止写文件与触发 Supervisor 生效命令，因此初始化方案只能是“读取远端 + 写数据库”。

## 业务规则变更

- `sys_supervisor_service` 扩展为统一模型，同时承载：
  - `TEMPLATE_MANAGED`：本地主机模板纳管记录
  - `IMPORTED_READONLY`：远端初始化导入快照
- 新增字段：
  - `config_path`：相对 `/etc/supervisord.d` 的相对路径
  - `file_name`：配置文件 basename
  - `content_program_name`：配置内容 `[program:*]` 中的名称
  - `manage_mode`：纳管模式
  - `baseline_content`：模板基线或导入原文快照
  - `metadata_complete`：结构化字段是否完整
  - `parse_warnings`：解析告警
- 字段冗余规则固定为：
  - `file_name` 与 `content_program_name` 永远独立保存
  - `config_name = file_name`
  - `program_name = content_program_name`
  - 禁止再用文件名反推 `program_name`，也禁止反向覆盖文件名
- 真实唯一键改为 `(host_ip, config_path)`。
- 远端导入按 `config_path` 读取和定位文件，不再按 `config_name` 拼接顶层路径。
- 远端导入解析采用宽松模式：
  - 允许 `[program: xxx]` 中冒号后有空格
  - 允许重复 INI key，按最后一个值生效
  - 结构化字段能提取就填，提取不到就留空
  - 只要能识别为单个合法 `[program:*]` 段就允许入库

## 接口或配置变更

### 接口返回

列表和详情接口新增以下字段：

- `configPath`
- `fileName`
- `contentProgramName`
- `manageMode`
- `metadataComplete`
- `parseWarnings`

兼容字段继续保留：

- `configName = fileName`
- `programName = contentProgramName`

`GET /admin/api/supervisor/services/{programName}` 仍按 `program_name` 查询；当前实现通过业务校验保持同主机下 `program_name` 不重复。

### 导入脚本

导入脚本改为显式工具模式：

- `--host 10.1.0.104`：仅导入指定主机
- `--recursive`：递归扫描 `/etc/supervisord.d` 子目录
- 默认 `dry-run`
- 只有 `--apply` 才会真正写入数据库

### 数据库基线

当前这些字段和索引已经直接并入 `app/database/migrations/001_init_schema.sql` 单基线初始化 SQL，不再依赖额外的增量 `ALTER TABLE` 迁移来补齐结构。

## 对调用方/运维的影响

- `GET /services` 返回空数组时，需要先确认数据库是否已导入主数据，而不是继续假设远端没有服务。
- 远端只读主机的详情页如果 `manageMode=IMPORTED_READONLY`，`expectedContent` 来自 `baseline_content`，不再强制按当前模板重渲染。
- 远端导入记录允许存在“不完整但可识别”的结构化数据，调用方要结合 `metadataComplete` 与 `parseWarnings` 判断是否适合后续人工治理。
- 运维执行初始化导入时，推荐先用 dry-run 验证范围和跳过原因，再决定是否追加 `--apply`。

## 验证方式

- 运行 `.venv/bin/python -m compileall app tests scripts/import_supervisor_services.py`
- 运行 `PYTHONPATH=. .venv/bin/pytest -q`
- dry-run 验证：
  - `.venv/bin/python scripts/import_supervisor_services.py --host 10.1.0.104 --recursive`
- apply 验证：
  - `.venv/bin/python scripts/import_supervisor_services.py --host 10.1.0.104 --recursive --apply`
- 验收重点：
  - `configPath/fileName/contentProgramName` 能同时返回
  - `fileName` 与 `contentProgramName` 不一致时仍原样保留
  - 远端服务器文件无任何新增、删除、覆盖
  - 不执行任何 `reread/update/start/stop/restart`
