# 跨域预检与跨源访问说明

## 背景与目标

前端管理后台通常与后端分离部署，需要稳定支持浏览器跨源调用和 `OPTIONS` 预检。

## 当前问题或修改原因

如果把跨域预检继续落到鉴权或业务路由，会导致浏览器在真正发起业务请求前就失败。

## 业务规则变更

- 统一只对 `/admin/api/*` 补齐 CORS 响应头
- `OPTIONS` 请求直接返回 `200`
- `OPTIONS` 不做 JWT 校验
- 真正的业务请求仍然要求 Bearer JWT

当前固定响应头：

- `Access-Control-Allow-Origin`
- `Access-Control-Allow-Credentials`
- `Access-Control-Allow-Methods`
- `Access-Control-Allow-Headers`
- `Access-Control-Expose-Headers`

## 接口或配置变更

- 无新增接口
- 由应用中间件统一处理跨域逻辑

## 对调用方/运维的影响

- 前端无需为每个 API 单独处理跨域逻辑
- 只要请求路径位于 `/admin/api/*`，浏览器预检都应稳定通过

## 验证方式

- 浏览器跨源访问 `GET/POST/PUT/DELETE`
- 手工发送 `OPTIONS /admin/api/...`
- 确认预检不返回 `401`
