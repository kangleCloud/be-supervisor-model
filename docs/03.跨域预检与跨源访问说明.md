# 跨域预检与跨源访问说明

## 背景与目标

`be-supervisor-model` 已经提供前端管理后台需要的登录鉴权与 Supervisor 管理接口，但此前默认只考虑同源或脚本直连调用，没有补齐浏览器跨源访问所需的 CORS 响应头。

本次修改的目标是：

- 让 `fe-supervisor-model` 等浏览器前端可以直接跨源访问 `/admin/api/*`
- 对齐 `be-vita` 的跨域处理策略，避免前端在 `Authorization` 与 JSON 请求场景下被预检阻断
- 不引入新的配置项，不改变现有接口路径、请求结构和响应结构

## 当前问题或修改原因

浏览器里显示的 `strict-origin-when-cross-origin` 是默认的 Referrer Policy，并不是本次请求失败的直接原因。

真正的问题在于：

- 前端带 `Authorization` 请求头访问受保护接口时，浏览器会先发送 `OPTIONS` 预检
- 前端以 `application/json` 提交登录或新增服务时，同样会触发 `OPTIONS` 预检
- 服务端此前没有统一处理 `OPTIONS`，也没有返回 `Access-Control-Allow-*` 响应头，导致浏览器直接拦截正式请求

## 业务规则变更

- 所有 `/admin/api/*` 请求统一补齐跨域响应头
- `OPTIONS` 预检请求直接返回 `200`，不进入 JWT 校验、参数校验和业务路由
- `Access-Control-Allow-Origin` 按请求头 `Origin` 原样回写；如果请求没有 `Origin`，则返回 `*`
- 继续允许浏览器携带凭证语义，返回 `Access-Control-Allow-Credentials: true`
- 固定放行 `GET,POST,PUT,DELETE,OPTIONS` 方法
- 固定放行 `Authorization, Content-Type, Accept, X-Requested-With, Cache-Control, Pragma` 请求头
- 对调用方暴露 `Authorization` 响应头

## 接口或配置变更

接口层面：

- `OPTIONS /admin/api/auth/login` 新增可直接成功返回的预检行为
- `OPTIONS /admin/api/supervisor/*` 新增可直接成功返回的预检行为
- 受保护接口即使返回 `401/400/500`，只要是 `/admin/api/*`，也会携带统一的 CORS 响应头

配置层面：

- 不新增 `config.yaml` 配置项
- 不新增环境变量
- 不调整 JWT、主机白名单、执行器和 Supervisor 配置逻辑

## 对调用方/运维的影响

- 前端开发环境可直接从 `http://127.0.0.1:5173` 或其他跨源地址访问后端接口，无需额外代理才能通过浏览器预检
- 调用方仍需按原规则携带 `Authorization: Bearer <jwt>`，跨域放行不代表跳过鉴权
- 运维侧无需增加新的部署参数；如果后续需要更严格的生产域名限制，再单独演进为白名单配置方案

## 验证方式

自动化验证：

- `OPTIONS /admin/api/auth/login` 返回 `200` 且携带完整 CORS 响应头
- `OPTIONS /admin/api/supervisor/hosts` 返回 `200` 且不会触发未登录 `401`
- 未登录访问 `GET /admin/api/supervisor/hosts` 时仍返回 `401`，同时带 CORS 响应头
- 已登录访问受保护接口成功时，同样带 CORS 响应头

手工验证：

- 启动 `fe-supervisor-model` 与 `be-supervisor-model`
- 在浏览器中执行登录、读取当前用户、查询主机列表
- 确认 Network 面板里预检请求与正式请求都成功返回
