# Supervisor 导入初始化 API 约定

## 背景与目标

前端和脚本统一通过同一套两阶段导入协议完成现网配置初始化接入。

## 当前问题或修改原因

导入扫描耗时较长，且需要支持页面刷新后恢复预检结果，因此必须把导入拆成暂存恢复、预检和提交三步。

## 业务规则变更

- 仅保留两种导入模式：
  - `PRECHECK`
  - `COMMIT`
- 预检结果先写暂存表
- 正式提交按批次原子写主表
- 成功后清理暂存，失败保留暂存

## 接口或配置变更

### 1. 恢复暂存

`GET /admin/api/supervisor/imports/staging?host=...`

返回：

- `exists`
- `batchId`
- `createdAt`
- `summary`
- `items`

### 2. 预检

`POST /admin/api/supervisor/imports`

```json
{
  "host": "10.1.0.104",
  "mode": "PRECHECK"
}
```

### 3. 提交

`POST /admin/api/supervisor/imports`

```json
{
  "host": "10.1.0.104",
  "mode": "COMMIT",
  "batchId": "..."
}
```

约束：

- `PRECHECK` 禁止携带 `batchId`
- `COMMIT` 必须携带 `batchId`
- 结果中存在 `SKIPPED` 时，`COMMIT` 返回 `409`
- 扫描范围固定为 `/etc/supervisord.d/**/*.ini`

## 对调用方/运维的影响

- 前端打开弹窗、切换主机、刷新页面时应先调 `GET staging`
- 只有 `planned > 0 && skipped == 0` 时才应允许点击“导入”
- 脚本模式默认等价于 `PRECHECK`，加 `--apply` 后再顺序执行 `COMMIT`

## 验证方式

- `PRECHECK` 能写暂存且不写主表
- `GET staging` 能恢复最近批次
- `COMMIT` 成功后写主表并清暂存
- `COMMIT` 失败时正式表回滚、暂存保留
