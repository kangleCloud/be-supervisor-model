# Ansible 输出兼容与告警治理

## 背景与目标

远端操作大量依赖 ansible ad-hoc 命令，必须稳定提取真实 stdout/stderr，并把 warning 噪音与真正错误区分开。

## 当前问题或修改原因

不同 callback 输出格式、stderr warning 和 inventory 配置错误都可能污染接口诊断，如果不统一清洗，前端和运维很难快速定位问题。

## 业务规则变更

- 固定使用 `ansible.builtin.minimal` callback
- stdout 解析统一提取真实业务负载
- stderr 统一剥离 warning/deprecation 噪音
- `inventoryPath` 缺失或不可读时，优先返回控制机配置错误
- `host pattern` 未匹配时返回“目标主机未匹配”
- SSH/超时/连接失败时统一归类为“目标主机不可达”

## 接口或配置变更

- 概况、导入、同步和远端变更都复用 `executor.timeoutSeconds`
- 不再为概况接口保留独立短超时

## 对调用方/运维的影响

- 前端应展示接口返回的 `warnings` 或错误信息，不应自行拼装失败原因
- 运维需要优先检查 inventory、SSH 连通性和远端命令环境

## 验证方式

- inventory 不存在时返回明确配置错误
- SSH 超时时返回不可达
- 成功场景不会把 stderr warning 当成接口失败
