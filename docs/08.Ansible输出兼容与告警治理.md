# Ansible 输出兼容与告警治理

## 背景与目标

远端只读链路当前依赖 `ansible` ad-hoc 命令读取配置目录、配置文件和 `supervisorctl status`。随着控制机升级到 `ansible-core 2.21.x`，历史使用的 `ansible -o` 和 `oneline` 输出语义已经进入弃用路径，直接把 deprecation warning 暴露到了业务错误文案里。

本次修改目标是移除弃用依赖，保证远端读取仍然只读，同时让接口在成功场景下不再被 warning 污染，失败场景能优先返回真实根因。

## 当前问题或修改原因

- `AnsibleExecutor` 仍然硬编码 `ansible ... -o`，会持续触发 deprecation warning。
- 旧版 stdout 解析主要围绕 `(stdout)` 形态实现，无法稳定表达 `minimal/default` 多行结果头。
- `stderr` 中的 warning 与真实错误没有分层，导致调用方看到的“读取配置目录失败”可能只剩 warning，而看不到真正的目录缺失、SSH 不可达或 inventory 未匹配。

## 业务规则变更

- `ansible` 执行器不再传递 `-o` 参数。
- 服务端固定通过 `ANSIBLE_STDOUT_CALLBACK=ansible.builtin.minimal` 读取业务 stdout。
- stdout 解析统一兼容两类结果：
  - `minimal/default` 的 `host | CHANGED | rc=0 >>` 多行正文
  - 历史 `(stdout)` 形态
- stderr 清洗规则固定为：
  - `exit_code == 0` 时，warning/deprecation 噪音不参与业务失败判定
  - `exit_code != 0` 时，先剥离 warning/deprecation 噪音，再把真实错误透传给上层
  - inventory 未匹配仍然优先改写为统一中文错误文案
- warning 只记录到服务端日志或内部调试信息，不进入 HTTP 响应体。

## 接口或配置变更

- HTTP 路由、请求参数、响应字段不变。
- 对调用方可感知的变化只有错误质量：
  - 成功场景不再因为 deprecation warning 被误判失败
  - 失败场景优先看到真实根因，而不是 callback 弃用提示
- 不新增 `ansible.cfg` 级别的 `deprecation_warnings=False` 配置；根修优先于静默压制。

## 对调用方/运维的影响

- 前端和调用方不需要调整接口契约。
- 运维排查远端读取问题时，应重点关注真正的目录缺失、inventory 未匹配、SSH 不可达等根因，不再把 `-o/oneline` warning 当成业务故障。
- 所有 `executorType=ansible` 的白名单主机统一应用这套修复，不只针对 `10.1.0.252`。

## 验证方式

- 运行 `.venv/bin/python -m compileall app tests`
- 运行 `PYTHONPATH=. .venv/bin/pytest -q`
- 核心验收点：
  - 结果命令参数中不再包含 `-o`
  - `minimal` 多行 stdout 可以正确解析
  - 成功场景即使 stderr 有 warning，列表/详情/导入链路仍按成功返回
  - 失败场景若 stderr 同时包含 warning 和真实错误，只返回真实错误
  - inventory 未匹配仍返回统一中文错误文案
