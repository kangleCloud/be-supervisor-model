# Supervisor 归档与运行操作联动说明

## 背景与目标

此前仓库只支持 Supervisor 配置读取、初始化导入和新增，不支持远端运行操作，也不支持把数据库中的归档动作同步到远端 `.ini` 现场。

本次调整的目标是：

- 为前端提供统一的运行操作接口：启动、停止、重启
- 为纳管服务提供归档/还原接口，并联动远端配置文件现场
- 保持“新增服务仅限 local 主机”的边界不变
- 把列表、详情、运行操作和归档状态统一到数据库主数据语义上

## 当前问题或修改原因

- 远端 `ansible` 主机原先被一刀切限制为“只能读”，导致已纳管服务无法通过后台执行常见运维动作。
- 数据库没有显式的归档字段，服务列表无法区分“在线纳管”与“历史归档”。
- 归档如果只改数据库、不改远端 `.ini`，会让 Supervisor 现场和系统视图长期漂移。
- 导入记录可能位于 `/etc/supervisord.d` 子目录，旧的顶层 `configName` 写法不足以安全定位归档/还原目标。

## 业务规则变更

- `sys_supervisor_service` 新增：
  - `is_archived`
  - `archived_at`
  - `restored_at`
- 服务列表新增 `archived=false|true|all` 查询语义，默认 `false`，只展示未归档记录。
- 远端 `ansible` 主机放开以下能力：
  - `supervisorctl start`
  - `supervisorctl stop`
  - `supervisorctl restart`
  - 归档/还原链路中的 `reread/update`
  - 按数据库 `configPath` 备份、删除、恢复远端 `.ini`
- 新增服务仍然只允许 `local` 主机，不允许通过 API 在远端写入全新模板配置。
- 归档固定流程：
  1. 校验记录存在且未归档
  2. 先 `stop`
  3. 备份 `.ini` 为 `.bak`
  4. 删除 `.ini`
  5. `reread`
  6. `update`
  7. 数据库写 `is_archived=true`
- 还原固定流程：
  1. 校验记录存在且已归档
  2. 从 `.bak` 恢复 `.ini`
  3. `reread`
  4. `update`
  5. 查询单服务状态并回写数据库
  6. 不自动启动服务
- 已归档服务禁止执行 `start/stop/restart`，后端直接返回业务错误，不依赖前端按钮隐藏兜底。

## 接口或配置变更

新增接口：

- `POST /admin/api/supervisor/services/{programName}/start?host=...`
- `POST /admin/api/supervisor/services/{programName}/stop?host=...`
- `POST /admin/api/supervisor/services/{programName}/restart?host=...`
- `POST /admin/api/supervisor/services/{programName}/archive?host=...`
- `POST /admin/api/supervisor/services/{programName}/restore?host=...`

既有接口变化：

- `GET /admin/api/supervisor/services`
  - 新增查询参数 `archived`
  - 分页记录新增 `isArchived`、`archivedAt`、`restoredAt`
- `GET /admin/api/supervisor/services/{programName}`
  - 详情新增 `isArchived`、`archivedAt`、`restoredAt`

迁移变更：

- 归档相关字段和索引由 Tortoise ORM 模型定义，并通过仓库根目录 `migrations/models/` 下的 Aerich migration 管理
- `app/database/migrations/001_init_schema.sql` 仅保留为历史参考快照，不再参与运行时初始化

## 对调用方/运维的影响

- 前端服务列表默认只会看到未归档记录；若需查看历史记录，必须显式传 `archived=true` 或 `archived=all`。
- 归档后的服务详情仍可查询，但运行按钮应隐藏或禁用。
- 还原成功后不会自动启动服务，前端如需恢复运行，必须单独调用启动接口。
- 运维侧若存在子目录配置，归档/还原会严格按数据库中的 `configPath` 执行，不再假设配置一定在顶层目录。
- 导入脚本和导入 API 仍然只负责数据库初始化，不会参与归档/还原现场改动。

## 验证方式

执行以下命令：

```bash
python3 -m compileall app tests scripts
.venv/bin/python -m pytest -q
```

重点验证：

- 新库初始化包含归档字段，旧库可通过 `003` 补齐
- 列表默认只返回未归档记录，`archived=true|all` 语义正确
- 远端 `ansible` 主机可执行 `start/stop/restart`
- 归档会联动 `.bak` 备份、删除 `.ini` 并刷新数据库归档状态
- 还原会恢复 `.ini`，刷新数据库状态，但不会自动启动服务
