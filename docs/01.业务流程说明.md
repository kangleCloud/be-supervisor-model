# be-supervisor-model 业务流程说明

## 1. 背景与目标

`be-supervisor-model` 是一个面向运维后台的管理服务，不承担业务系统本身的业务逻辑。

当前系统聚焦两类能力：

- 登录鉴权：提供登录、当前登录态查询、退出登录，并把登录令牌与登录审计写入 MySQL
- Supervisor 管理：纳管目标主机上的 Supervisor 服务配置，并提供读取、初始化导入、新增、修改、删除、运行操作和归档/还原接口

系统目标是为后续前端管理后台或内部运维工具提供统一的 HTTP 接口，用于：

- 限定白名单主机范围
- 安全生成 `/etc/supervisord.d` 下的 Supervisor 配置
- 将配置主数据落库，避免列表长期依赖远端全量扫目录
- 在本地与远端主机场景执行受控的配置写入、配置删除和 `supervisorctl` 操作
- 把读取、同步、创建、修改、删除、归档还原全部收敛到数据库主数据语义

## 2. 角色与职责

| 角色/组件 | 职责 |
| --- | --- |
| 运维管理员 | 登录后台后查询 Supervisor 服务、执行初始化导入、对本地或远端服务执行新增/修改/删除、运行操作和归档还原 |
| `be-supervisor-model` | 统一处理登录鉴权、主机白名单校验、配置模板渲染、初始化导入、受控文件写入与删除、远端受控命令、归档联动和主数据落库 |
| MySQL 8 | 存储用户、JWT 登录令牌、登录审计和 Supervisor 服务主数据 |
| `local` 执行器 | 当管理服务与目标应用同机部署时，直接在本机执行文件操作和 `supervisorctl` |
| `ansible` 执行器 | 当目标主机为远程机器时，通过 `ansible` ad-hoc 命令执行远端配置读取、写入、删除和 `supervisorctl` 命令 |
| 被管理主机 | 提供 `/etc/supervisord.d` 配置目录和 `supervisorctl` 命令环境 |

## 3. 认证与令牌流程

### 3.1 认证接口

认证接口统一挂载在：

```text
/admin/api/auth/*
```

当前提供：

- `POST /admin/api/auth/login`
- `GET /admin/api/auth/profile`
- `POST /admin/api/auth/logout`
- 所有 `/admin/api/*` 在跨源访问时统一由服务端补齐 CORS 响应头，`OPTIONS` 预检请求不做登录校验

### 3.2 登录链路

1. 运维管理员调用 `/admin/api/auth/login`，提交用户名和密码。
2. 服务从 `sys_user` 表读取账号信息，校验账号存在、未删除、状态为启用。
3. 服务校验密码哈希。
4. 校验通过后签发 JWT 访问令牌。
5. 服务把登录令牌写入 `sys_login_token`，只保存 `token_jti` 和 `token_digest`，不保存原始 token 明文。
6. 服务更新 `sys_user` 的最近登录时间和登录地址。
7. 服务把本次成功或失败登录写入 `sys_login_log`。
8. 调用方后续访问受保护接口时，统一通过 `Authorization: Bearer <jwt>` 传递访问令牌。

### 3.3 令牌校验与退出

1. 受保护接口统一校验 Bearer JWT。
2. 服务先校验 JWT 签名和过期时间，再根据 `sub + jti` 到 `sys_login_token` 查询活动令牌。
3. 令牌存在且 `token_digest` 匹配时，才允许继续访问。
4. `/admin/api/auth/logout` 会把当前令牌写入 `revoked_time`，使同一 JWT 后续请求立即失效。

### 3.4 认证相关数据表

当前登录鉴权依赖以下表：

- `sys_user`
- `sys_login_token`
- `sys_login_log`

数据库启动规则：

- 服务启动只初始化/关闭 Tortoise ORM 连接，不再自动建库、建表或执行 SQL 初始化
- 业务库必须由外部环境预先创建
- schema 变更统一通过 Aerich 执行，仓库根目录 `migrations/models/` 为唯一迁移来源
- `app/database/migrations/001_init_schema.sql` 仅保留为历史 DDL 参考快照，不参与运行时链路
- 默认超级管理员 `admin / Admin@123456` 由 Aerich baseline 一并落库，不再依赖应用启动补种

## 4. Supervisor 管理流程

### 4.1 Supervisor 接口

Supervisor 管理接口统一挂载在：

```text
/admin/api/supervisor/*
```

当前提供：

- `GET /admin/api/supervisor/hosts`
- `GET /admin/api/supervisor/overview`
- `POST /admin/api/supervisor/imports`
- `GET /admin/api/supervisor/services`
- `GET /admin/api/supervisor/services/{programName}`
- `POST /admin/api/supervisor/services/{programName}/sync`
- `POST /admin/api/supervisor/services`
- `PUT /admin/api/supervisor/services/{programName}`
- `DELETE /admin/api/supervisor/services/{programName}`
- `POST /admin/api/supervisor/services/{programName}/start`
- `POST /admin/api/supervisor/services/{programName}/stop`
- `POST /admin/api/supervisor/services/{programName}/restart`
- `POST /admin/api/supervisor/services/{programName}/archive`
- `POST /admin/api/supervisor/services/{programName}/restore`
- 前端跨源调用上述接口时，浏览器预检会被服务端直接放行，不进入业务路由

### 4.2 主数据来源

Supervisor 管理采用“数据库为主、远端文件为执行结果”的模型：

- `sys_supervisor_service`：统一保存模板纳管记录和远端只读导入快照
- `/etc/supervisord.d/**/*.ini`：保存目标主机现场的实际 Supervisor 配置
- `supervisorctl status`：仅在显式同步或运行操作回读时查询远端状态

接口职责固定为：

- 主机概况：实时读取白名单主机 CPU / 内存 / 基础检查结果，不写数据库
- 服务列表：只查 `sys_supervisor_service` 分页快照
- 服务详情：只查 `sys_supervisor_service` 单条快照
- 单服务同步：显式读取远端 `supervisorctl status`、当前 `.ini` 和可选 `.bak`，再把结果写回 `sys_supervisor_service`

当前表内字段语义固定为：

- `config_path`：相对 `/etc/supervisord.d` 的相对路径，是现场真实定位键
- `file_name`：配置文件 basename
- `content_program_name`：配置内容 `[program:*]` 中的名称
- `config_name`：兼容展示字段，固定等于 `file_name`
- `program_name`：兼容展示字段，固定等于 `content_program_name`
- `manage_mode`：`TEMPLATE_MANAGED` 或 `IMPORTED_READONLY`
- `baseline_content`：模板基线或导入时保存的原始配置文本
- `is_archived / archived_at / restored_at`：归档状态和时间戳

### 4.3 主机白名单与执行模型

目标主机必须来自配置中的 `hosts` 白名单，不允许调用方传入任意执行目标。

每台主机都绑定一种执行方式：

- `local`：直接在本机操作配置与执行命令
- `ansible`：通过 `ansible` 对远程主机执行读取、模板写入/删除、运行操作与归档联动

远端执行输出规则：

- `ansible` 执行器固定使用受支持的 `minimal` callback 读取业务 stdout
- 不再依赖已弃用的 `ansible -o` / `oneline` 输出语义
- `FAILED/UNREACHABLE` 结果头里的失败正文会被提取出来，失败时优先返回真实根因
- Ansible 自身的 warning/deprecation 噪音只在服务端日志或内部调试中处理，不向接口响应透传

执行器仅暴露受控能力：

- 列出配置文件
- 读取配置文本
- 对目标主机原子写入配置文件
- 对已纳管记录按 `configPath` 执行备份、删除和恢复
- 执行监听端口检查命令
- 执行固定格式的 `supervisorctl` 命令
- 执行固定只读的主机概况采集脚本（`sh -lc`），读取 `/proc`、`hostname` 和受控检查项
- 远端 ansible 调用前先校验控制机 `inventoryPath` 可读；概况、导入、同步等读取能力共用同一套 ansible 基础配置与超时

当前项目强约束：

- 本地主机和 `executorType=ansible` 的远端主机都允许创建、修改、删除未归档模板服务
- 删除固定先 `stop`，再删除当前 `.ini` 和数据库记录，但保留 `.bak`
- 已归档记录禁止直接修改或删除，必须先还原
- Supervisor 页面展示数据统一以接口为唯一来源，不允许前端用 mock 数据回填概况、检查项或列表状态
- `executorType=ansible` 的远端主机允许读取配置、执行 `start/stop/restart`，以及按已纳管 `configPath` 归档/还原
- 远端主机统一以 IP 作为唯一标识，`ansible` 执行器固定使用 `host.ip` 作为 inventory pattern，不再支持逻辑别名

配置文件读取来源取决于执行器与部署位置：

- `local` 场景下，配置文件直接从本机读取
- `ansible` 场景下，配置文件通过远程服务器读取

### 4.4 配置文件目录与命名规则

Supervisor 配置目录固定为：

```text
/etc/supervisord.d
```

服务端对该路径只做词法归一化，不做宿主机物理 `resolve()`，避免在 macOS 控制机上把 `/etc/...` 错误改写成 `/private/etc/...` 后继续传到远端 Linux。

受控扫描范围包括：

- `*.ini`
- `*.ini.bak`
- `*.ini.bak.*`

只读导入和端口扫描默认允许递归子目录；模板新增仍固定写入 `/etc/supervisord.d` 顶层；若修改的是子目录记录，则保留原父目录，只替换 basename。

默认命名规则：

- `programName = {jobName}_{moduleName}`
- `configName = {programName}.ini`

现网导入新增的冗余规则：

- `fileName` 与 `contentProgramName` 永远独立保存
- 即使 `fileName != contentProgramName + ".ini"` 也允许入库
- 系统禁止再用文件名反推 `programName` 覆盖现场原值，也禁止反向覆盖文件名

服务端负责：

- 校验 `jobName / moduleName / programName / configName` 安全正则
- 拒绝 `../`、绝对路径注入和备份文件名伪装
- 把配置文件路径限制在 `/etc/supervisord.d`

### 4.5 模板渲染规则

当前创建/修改接口允许调用方输入的主要字段为：

- `jobName`
- `moduleName`
- `javaPath`
- `active`
- `port`
- `jarName`
- `configName`
- `xms`
- `xmx`
- `user`

说明：

- `POST /admin/api/supervisor/services` 通过请求体传 `host`
- `PUT /admin/api/supervisor/services/{programName}` 通过查询参数 `host` 定位旧记录，请求体不再重复传 `host`

以下 Supervisor 模板参数固定为系统内建策略，不接受前端自定义输入：

- `autostart=true`
- `startsecs=10`
- `autorestart=true`
- `startretries=3`
- `priority=999`
- `redirect_stderr=true`
- `stdout_logfile_maxbytes=1GB`
- `stdout_logfile_backups=1`
- `stopasgroup=false`
- `killasgroup=false`

### 4.6 新增、修改、删除服务流程

创建流程：

1. 校验 Bearer JWT 与令牌状态。
2. 校验目标主机必须在白名单中，允许 `local + ansible`。
3. 根据请求变量生成 `programName`、`configName` 和目标 `configPath`。
4. 先校验数据库中同主机下的 `programName / configPath / port` 不能重复。
5. 检查目标配置文件是否已存在。
6. 递归扫描 `/etc/supervisord.d` 下的 `.ini/.ini.bak/.ini.bak.*`，检测文件端口冲突。
7. 执行 `ss -lnutp` 检查目标端口是否已被监听。
8. 按固定模板渲染 Supervisor 配置。
9. 原子写入目标配置文件。
10. 执行 `supervisorctl reread`。
11. 执行 `supervisorctl update`。
12. 现场成功后，把模板纳管记录写入 `sys_supervisor_service`，同时保存 `configPath/fileName/contentProgramName/manageMode/baselineContent`。
13. 如果写库失败，立即删除刚写入的 `.ini`，再执行一次 `reread/update` 回滚现场。

修改流程：

1. 通过 `host + 路径中的旧 programName` 查询数据库记录。
2. 若记录不存在或已归档，则直接拒绝。
3. 重新按请求字段渲染目标模板，并生成新的 `programName/configName/configPath`。
4. 以“可排除当前记录”的方式校验数据库唯一性。
5. 固定执行文件冲突校验；若端口发生变化，再额外执行 `ss -lnutp` 监听冲突校验。
6. 若 `programName` 或 `configPath` 发生变化：
   - 先执行 `supervisorctl stop`，允许原本未运行
   - 备份旧 `.ini`
   - 写入新 `.ini`
   - 删除旧 `.ini`
7. 若 `programName` 和 `configPath` 未变化：
   - 先备份当前 `.ini`
   - 再覆盖写入当前路径
8. 执行 `supervisorctl reread/update`。
9. 更新数据库主数据、运行状态快照和详情同步快照字段。
10. 如果原记录是 `IMPORTED_READONLY`，修改成功后固定转成 `TEMPLATE_MANAGED`。
11. 如果写库失败，优先从 `.bak` 恢复旧配置，再执行一次 `reread/update` 回滚现场。

删除流程：

1. 通过 `host + programName` 查询数据库记录。
2. 若记录不存在则返回 `404`；若已归档则直接拒绝。
3. 固定先执行 `supervisorctl stop`，允许“未运行”或“已无此进程”继续删除。
4. 如果当前 `.ini` 存在，则先刷新或创建 `.bak`，再删除当前 `.ini`。
5. 如果当前 `.ini` 不存在，不强制失败，直接继续后续流程。
6. 执行 `supervisorctl reread/update`。
7. 删除数据库记录。
8. 如果删库失败，优先从 `.bak` 恢复 `.ini`，再执行一次 `reread/update` 回滚现场。

### 4.7 服务读取与单服务同步流程

服务读取固定拆成两条链路：

- 列表与详情链路：默认只读数据库，不触发任何远端命令
- 单服务同步链路：显式读取远端状态与配置，然后把同步结果回写数据库

服务列表接口返回：

- 结构化主数据：`configName`、`programName`、`configPath`、`fileName`、`contentProgramName`
- 纳管与解析字段：`manageMode`、`metadataComplete`、`parseWarnings`
- 运行状态快照：`status`、`pid`、`uptime`
- 归档字段：`isArchived`、`archivedAt`、`restoredAt`

服务详情接口返回：

- 基础字段：`id`、`host`、`hostName`、`programName`、`configName`、`configPath`
- 结构化字段：`jobName`、`moduleName`、`javaPath`、`active`、`port`、`jarName`、`xms`、`xmx`、`user`
- 最近同步快照：`command`、`directory`、`stdoutLogfile`、`status`、`pid`、`uptime`
- 配置快照：`hasBackup`、`configContent`、`backupConfigContent`
- 同步元数据：`lastSyncAt`、`syncStatus`、`syncError`
- 归档字段：`isArchived`、`archivedAt`、`restoredAt`

读取要求：

- 本地主机和远端主机都允许执行列表、详情和初始化导入 API 的只读读取
- 服务列表默认只查询 `is_archived = false`；调用方可通过 `archived=true|all` 查看归档记录
- 打开详情页时不再隐式远端读取；前端如需最新现场信息，必须显式调用同步接口

单服务同步流程：

1. 校验 JWT、主机白名单、数据库记录存在且 `is_archived=false`。
2. 按数据库 `programName` 执行 `supervisorctl status {program}`。
3. 按数据库 `configPath` 读取当前 `.ini`。
4. 检查并按需读取 `configPath + ".bak"`。
5. 复用配置解析器提取 `command/directory/stdout_logfile` 与结构化字段。
6. 成功时把远端状态和配置快照写回数据库，详情后续直接读库。
7. 若主配置缺失或状态条目缺失，同步结果写为 `FAILED`，但仍尽量保留已同步到的部分字段。
8. 已归档服务默认禁止同步远端详情。

### 4.8 运行操作与归档联动流程

运行操作规则：

1. `start/stop/restart` 在执行前统一校验 JWT、主机白名单、数据库记录存在且 `is_archived=false`。
2. 运行命令成功后，服务端会立即执行一次单服务 `supervisorctl status {program}`，把 `status/pid/uptime/status_sync_time` 刷回数据库。
3. 如果单服务状态回读失败：
   - `start/restart` 回写 `UNKNOWN`
   - `stop` 回写 `STOPPED`，并清空 `pid/uptime`
4. 已归档服务执行 `start/stop/restart` 会直接返回业务错误，不允许前端绕过按钮控制强行触发。

归档流程：

1. 校验数据库记录存在且当前 `is_archived=false`。
2. 先执行 `supervisorctl stop`，允许“原本未运行”场景继续归档。
3. 按数据库中的 `configPath` 创建 `.bak` 备份。
4. 删除远端 `.ini` 现场。
5. 执行 `supervisorctl reread`。
6. 执行 `supervisorctl update`。
7. 数据库写入 `is_archived=true`、`archived_at=当前时间`，并把状态回写为 `STOPPED`。

还原流程：

1. 校验数据库记录存在且当前 `is_archived=true`。
2. 按数据库中的 `configPath` 从 `.bak` 恢复 `.ini`。
3. 执行 `supervisorctl reread`。
4. 执行 `supervisorctl update`。
5. 查询一次单服务状态，并把最新 `status/pid/uptime` 与 `restored_at` 写回数据库。
6. 还原不会自动执行 `start`；前端如需拉起服务，应在还原成功后单独调用启动接口。

### 4.9 导入现网配置

为了切换到“库为主”的读取模型，现网已有配置需要先执行一次初始化导入。前端优先调用 `POST /admin/api/supervisor/imports`，命令行脚本仅作为运维兜底入口。

导入规则：

- 前端打开导入弹窗、切换主机、页面刷新时，先调用 `GET /admin/api/supervisor/imports/staging?host=...` 恢复当前用户该主机最近一次预检结果。
- `GET staging` 无数据时返回空状态；有数据时返回 `exists=true + batchId + summary + items + createdAt`，前端可直接展示并决定是否显示“导入”按钮。
- `mode=PRECHECK` 先扫描远端并写入暂存表，返回 `batchId + summary + items`
- `PRECHECK` 每次都会重新扫描远端，并覆盖该用户该主机上一批暂存结果
- `mode=COMMIT` 必须携带上一次 `PRECHECK` 返回的 `batchId`，整批原子写入正式表
- `PRECHECK` 禁止携带 `batchId`，`COMMIT` 缺少 `batchId` 会直接返回 `400`
- `PRECHECK` 结果中只要存在任意 `SKIPPED`，后端 `COMMIT` 就会返回 `409`
- `COMMIT` 成功后删除该批次暂存；若提交失败或返回 `409`，正式表整批回滚且暂存保留，便于前端继续展示和重试
- 递归扫描 `/etc/supervisord.d` 与全部子目录
- 只导入活跃 `*.ini`
- 不导入 `.bak` 和归档备份
- 导入记录统一标记为 `IMPORTED_READONLY`
- 结构化字段能提取就填，提取不到就留空；`metadata_complete` 用于标记是否完整
- 允许 legacy 文件存在 `[program: xxx]` 空格与重复 key；重复 key 按最后一个值生效，并写入 `parse_warnings`
- 只要能识别为单个合法 `[program:*]` 配置，就允许入库；只有无法识别为单个合法 `[program:*]` 的文件才跳过
- 如果同路径已有 `TEMPLATE_MANAGED` 记录，正式导入允许覆盖为 `IMPORTED_READONLY`
- 初始化导入 API 固定递归扫描 `*.ini`；命令行脚本允许通过 `--recursive` 复用同一套服务层逻辑
- API 和导入脚本都允许通过 `ansible` 读取远端主机现有配置，但只读不改远端文件
- 导入前置阶段会把“inventory 未匹配 / 远端主机不可达 / 目录下无可用 `*.ini`”统一收敛为 `404`
- `hostname` 探测只用于调试输出；即使探测失败，只要后续扫描成功，导入流程仍继续

## 5. 配置与运行模型

### 5.1 配置来源

当前配置采用：

- `config.yaml`
- `.env.dev` / `.env.prod`
- 进程显式环境变量覆盖

最终优先级为：

- 进程显式环境变量
- `.env.dev` / `.env.prod`
- `config.yaml`
- 代码默认值

环境文件选择规则：

- `APP_ENV_FILE` 优先级最高，传入后直接按绝对路径加载指定环境文件
- 未设置 `APP_ENV_FILE` 时，`APP_ENV=dev` 加载 `.env.dev`
- 未设置 `APP_ENV_FILE` 时，`APP_ENV=prod` 加载 `.env.prod`
- `APP_ENV` 不应写在 `.env.dev/.env.prod` 内，应由 `./scripts/run.sh dev|prod` 或外部环境传入

### 5.2 运行环境

项目默认通过 `.venv` 环境运行。

- 本地开发默认先创建并激活 `.venv`
- 测试命令默认在 `.venv` 中执行
- 手工启动服务时，也默认使用 `.venv` 中已安装的依赖

### 5.3 关键配置项

`config.yaml` 负责的主配置：

- `app.host`
- `app.port`
- `app.logLevel`
- `database.host`
- `database.port`
- `database.name`
- `database.user`
- `auth.accessTokenExpireMinutes`

Supervisor 与执行器主配置：

- `supervisor.confDir`
- `supervisor.commandTimeoutSeconds`（默认 300，环境变量 `COMMAND_TIMEOUT_SECONDS` 可覆盖）
- `executor.type`
- `executor.inventoryPath`
- `executor.remoteUser`
- `executor.timeoutSeconds`（默认 300，环境变量 `ANSIBLE_COMMAND_TIMEOUT_SECONDS` 可覆盖）
- `hosts`

`.env.dev` / `.env.prod` 负责的敏感覆盖项：

- `DATABASE_PASSWORD`
- `JWT_SECRET`
- `APP_CONFIG_PATH`

## 6. 安全边界

当前系统的核心安全边界如下：

- 目标主机必须来自配置白名单
- 禁止任意命令执行
- 禁止路径穿越
- 所有配置文件操作必须限制在 `/etc/supervisord.d`
- `programName/jobName/moduleName/configName` 必须满足安全正则
- 所有 shell 参数必须经过受控构造，不能直接拼接用户输入
- 默认不记录敏感凭证
- CORS 仅解决浏览器跨源访问可达性，不替代 JWT 鉴权和主机白名单校验
