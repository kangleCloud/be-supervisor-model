# be-supervisor-model 业务流程说明

## 1. 背景与目标

`be-supervisor-model` 是一个面向运维后台的管理服务，不承担业务系统本身的业务逻辑。

当前系统聚焦两类能力：

- 登录鉴权：提供登录、当前登录态查询、退出登录，并把登录令牌与登录审计写入 MySQL
- Supervisor 管理：纳管目标主机上的 Supervisor 服务配置，并提供读取和新增接口

系统目标是为后续前端管理后台或内部运维工具提供统一的 HTTP 接口，用于：

- 限定白名单主机范围
- 安全生成 `/etc/supervisord.d` 下的 Supervisor 配置
- 将配置主数据落库，避免列表长期依赖远端全量扫目录
- 在本地主机场景执行受控的 `supervisorctl reread/update/status`
- 在远端主机场景只读查询配置与状态，不允许改远端现场

## 2. 角色与职责

| 角色/组件 | 职责 |
| --- | --- |
| 运维管理员 | 登录后台后查询 Supervisor 服务，并仅对本地主机新增服务 |
| `be-supervisor-model` | 统一处理登录鉴权、主机白名单校验、配置模板渲染、本地写文件、主数据落库和状态聚合 |
| MySQL 8 | 存储用户、JWT 登录令牌、登录审计和 Supervisor 服务主数据 |
| `local` 执行器 | 当管理服务与目标应用同机部署时，直接在本机执行文件操作和 `supervisorctl` |
| `ansible` 执行器 | 当目标主机为远程机器时，通过 `ansible` ad-hoc 命令执行只读配置读取与状态查询 |
| 被管理主机 | 提供 `/etc/supervisord.d` 配置目录和 `supervisorctl` 命令环境 |

## 3. 认证与令牌流程

### 3.1 认证接口

认证接口统一挂载在：

```text
/admin/api/auth/*
```

当前提供：

- `POST /admin/api/auth/login`
- `GET /admin/api/auth/profile`
- `POST /admin/api/auth/logout`
- 所有 `/admin/api/*` 在跨源访问时统一由服务端补齐 CORS 响应头，`OPTIONS` 预检请求不做登录校验

### 3.2 登录链路

1. 运维管理员调用 `/admin/api/auth/login`，提交用户名和密码。
2. 服务从 `sys_user` 表读取账号信息，校验账号存在、未删除、状态为启用。
3. 服务校验密码哈希。
4. 校验通过后签发 JWT 访问令牌。
5. 服务把登录令牌写入 `sys_login_token`，只保存 `token_jti` 和 `token_digest`，不保存原始 token 明文。
6. 服务更新 `sys_user` 的最近登录时间和登录地址。
7. 服务把本次成功或失败登录写入 `sys_login_log`。
8. 调用方后续访问受保护接口时，统一通过 `Authorization: Bearer <jwt>` 传递访问令牌。

### 3.3 令牌校验与退出

1. 受保护接口统一校验 Bearer JWT。
2. 服务先校验 JWT 签名和过期时间，再根据 `sub + jti` 到 `sys_login_token` 查询活动令牌。
3. 令牌存在且 `token_digest` 匹配时，才允许继续访问。
4. `/admin/api/auth/logout` 会把当前令牌写入 `revoked_time`，使同一 JWT 后续请求立即失效。

### 3.4 认证相关数据表

当前登录鉴权依赖以下表：

- `sys_user`
- `sys_login_token`
- `sys_login_log`
- `sys_schema_migration`

数据库启动规则：

- 服务启动时自动创建业务库（若不存在）
- 自动执行 `app/database/migrations/*.sql`
- 自动初始化一条默认超级管理员：`admin / Admin@123456`

## 4. Supervisor 管理流程

### 4.1 Supervisor 接口

Supervisor 管理接口统一挂载在：

```text
/admin/api/supervisor/*
```

当前提供：

- `GET /admin/api/supervisor/hosts`
- `GET /admin/api/supervisor/services`
- `GET /admin/api/supervisor/services/{programName}`
- `POST /admin/api/supervisor/services`
- 前端跨源调用上述接口时，浏览器预检会被服务端直接放行，不进入业务路由

### 4.2 主数据来源

Supervisor 管理采用“数据库为主、远端文件为执行结果”的模型：

- `sys_supervisor_service`：统一保存模板纳管记录和远端只读导入快照
- `/etc/supervisord.d/**/*.ini`：保存目标主机现场的实际 Supervisor 配置
- `supervisorctl status`：返回运行状态

列表和详情接口固定先查 `sys_supervisor_service`，再补充远端文件状态与运行状态，不再以“扫远端目录结果”作为主视图。

当前表内字段语义固定为：

- `config_path`：相对 `/etc/supervisord.d` 的相对路径，是现场真实定位键
- `file_name`：配置文件 basename
- `content_program_name`：配置内容 `[program:*]` 中的名称
- `config_name`：兼容展示字段，固定等于 `file_name`
- `program_name`：兼容展示字段，固定等于 `content_program_name`
- `manage_mode`：`TEMPLATE_MANAGED` 或 `IMPORTED_READONLY`
- `baseline_content`：模板基线或导入时保存的原始配置文本

### 4.3 主机白名单与执行模型

目标主机必须来自配置中的 `hosts` 白名单，不允许调用方传入任意执行目标。

每台主机都绑定一种执行方式：

- `local`：直接在本机操作配置与执行命令
- `ansible`：通过 `ansible` 对远程主机执行只读配置与状态查询

执行器仅暴露受控能力：

- 列出配置文件
- 读取配置文本
- 对本地主机原子写入配置文件
- 对本地主机删除文件
- 执行固定格式的 `supervisorctl` 命令

当前项目强约束：

- `executorType=ansible` 的远端主机只允许读取配置文件和查询 `supervisorctl status`
- 禁止通过服务端修改远端 `.ini/.bak` 文件
- 禁止通过服务端触发远端 `supervisorctl start/stop/restart/reread/update`

配置文件读取来源取决于执行器与部署位置：

- `local` 场景下，配置文件直接从本机读取
- `ansible` 场景下，配置文件通过远程服务器读取

### 4.4 配置文件目录与命名规则

Supervisor 配置目录固定为：

```text
/etc/supervisord.d
```

服务端对该路径只做词法归一化，不做宿主机物理 `resolve()`，避免在 macOS 控制机上把 `/etc/...` 错误改写成 `/private/etc/...` 后继续传到远端 Linux。

受控扫描范围包括：

- `*.ini`
- `*.ini.bak`
- `*.ini.bak.*`

只读导入和端口扫描默认允许递归子目录；本地主机新增写入仍固定落在 `/etc/supervisord.d` 顶层。

默认命名规则：

- `programName = {jobName}_{moduleName}`
- `configName = {programName}.ini`

现网导入新增的冗余规则：

- `fileName` 与 `contentProgramName` 永远独立保存
- 即使 `fileName != contentProgramName + ".ini"` 也允许入库
- 系统禁止再用文件名反推 `programName` 覆盖现场原值，也禁止反向覆盖文件名

服务端负责：

- 校验 `jobName / moduleName / programName / configName` 安全正则
- 拒绝 `../`、绝对路径注入和备份文件名伪装
- 把配置文件路径限制在 `/etc/supervisord.d`

### 4.5 模板渲染规则

当前新增接口允许调用方输入的主要字段为：

- `host`
- `jobName`
- `moduleName`
- `javaPath`
- `active`
- `port`
- `jarName`
- `configName`
- `xms`
- `xmx`
- `user`

以下 Supervisor 模板参数固定为系统内建策略，不接受前端自定义输入：

- `autostart=true`
- `startsecs=10`
- `autorestart=true`
- `startretries=3`
- `priority=999`
- `redirect_stderr=true`
- `stdout_logfile_maxbytes=1GB`
- `stdout_logfile_backups=1`
- `stopasgroup=false`
- `killasgroup=false`

### 4.6 新增服务流程

1. 校验 Bearer JWT 与令牌状态。
2. 校验目标主机必须在白名单中，且必须是 `local` 主机。
3. 如果目标主机为 `ansible`，接口直接返回 `403`，文案为“当前项目禁止修改远端配置文件”。
4. 根据请求变量生成 `programName` 和 `configName`。
5. 先校验数据库中同主机下的 `programName / configPath / port` 不能重复。
6. 检查目标配置文件是否已存在。
7. 递归扫描 `/etc/supervisord.d` 下的 `.ini/.ini.bak/.ini.bak.*`，检测端口冲突，防止库外人工文件撞端口。
8. 按固定模板渲染 Supervisor 配置。
9. 原子写入目标配置文件。
10. 执行 `supervisorctl reread`。
11. 执行 `supervisorctl update`。
12. 本地现场成功后，把模板纳管记录写入 `sys_supervisor_service`，同时保存 `configPath/fileName/contentProgramName/manageMode/baselineContent`。
13. 如果写库失败，立即删除刚写入的 `.ini`，再执行一次 `reread/update` 回滚现场。

### 4.7 服务读取流程

服务列表和详情接口会合并三部分信息：

- 数据库视角：`sys_supervisor_service` 中的结构化主数据
- 配置文件视角：按 `configPath` 读取远端真实文件，并结合模板基线或导入快照生成 `expectedContent`
- 运行状态视角：`supervisorctl status` 返回结果

列表和详情接口会同时返回：

- 兼容字段：`configName`、`programName`
- 显式字段：`configPath`、`fileName`、`contentProgramName`、`manageMode`、`metadataComplete`、`parseWarnings`

读取要求：

- 本地主机和远端主机都允许执行列表、详情和导入脚本的只读读取
- 远端目录不存在、远端路径不可达或远端读取失败时，接口应返回明确错误，不能静默表现成空列表或 `MISSING`

远端文件状态固定分为：

- `MATCH`：远端文件存在且与数据库期望内容完全一致
- `MISSING`：数据库有记录，但远端 `.ini` 不存在
- `MISMATCH`：远端 `.ini` 存在，但内容与数据库渲染结果不一致

期望内容规则：

- `TEMPLATE_MANAGED`：按数据库结构化字段重新渲染模板
- `IMPORTED_READONLY`：直接返回 `baseline_content`

### 4.8 导入现网配置

为了切换到“库为主”的读取模型，现网已有配置需要先执行一次导入脚本。

导入规则：

- 递归扫描 `/etc/supervisord.d` 与全部子目录
- 只导入活跃 `*.ini`
- 不导入 `.bak` 和归档备份
- 导入记录统一标记为 `IMPORTED_READONLY`
- 结构化字段能提取就填，提取不到就留空；`metadata_complete` 用于标记是否完整
- 允许 legacy 文件存在 `[program: xxx]` 空格与重复 key；重复 key 按最后一个值生效，并写入 `parse_warnings`
- 只要能识别为单个合法 `[program:*]` 配置，就允许入库；只有无法识别为单个合法 `[program:*]` 的文件才跳过
- 导入脚本允许通过 `ansible` 读取远端主机现有配置，但只读不改远端文件
- 导入脚本默认 `dry-run`，只有显式传入 `--apply` 才会写数据库

## 5. 配置与运行模型

### 5.1 配置来源

当前配置采用：

- `config.yaml`
- 环境变量覆盖

环境变量优先级高于 YAML。

### 5.2 运行环境

项目默认通过 `.venv` 环境运行。

- 本地开发默认先创建并激活 `.venv`
- 测试命令默认在 `.venv` 中执行
- 手工启动服务时，也默认使用 `.venv` 中已安装的依赖

### 5.3 关键配置项

应用与认证相关：

- `APP_HOST`
- `APP_PORT`
- `APP_LOG_LEVEL`
- `DATABASE_HOST`
- `DATABASE_PORT`
- `DATABASE_NAME`
- `DATABASE_USER`
- `DATABASE_PASSWORD`
- `JWT_SECRET`
- `ACCESS_TOKEN_EXPIRE_MINUTES`

Supervisor 与执行器相关：

- `SUPERVISOR_CONF_DIR`
- `COMMAND_TIMEOUT_SECONDS`
- `EXECUTOR_TYPE`
- `ANSIBLE_INVENTORY_PATH`
- `ANSIBLE_REMOTE_USER`

## 6. 安全边界

当前系统的核心安全边界如下：

- 目标主机必须来自配置白名单
- 禁止任意命令执行
- 禁止路径穿越
- 所有配置文件操作必须限制在 `/etc/supervisord.d`
- `programName/jobName/moduleName/configName` 必须满足安全正则
- 所有 shell 参数必须经过受控构造，不能直接拼接用户输入
- 默认不记录敏感凭证
- CORS 仅解决浏览器跨源访问可达性，不替代 JWT 鉴权和主机白名单校验
