# be-supervisor-model 业务流程说明

## 1. 背景与目标

`be-supervisor-model` 面向运维后台场景，负责把白名单主机上的 Supervisor 配置管理、状态同步和初始化导入统一收敛成 HTTP API。

系统目标：

- 限定主机范围，禁止任意目标执行
- 让服务列表和详情稳定以数据库为主
- 把远端读取、模板写入、运行操作与归档/删除流程标准化
- 为前端提供稳定的初始化导入、详情同步和概况刷新接口

## 2. 角色与职责

| 角色/组件 | 职责 |
| --- | --- |
| 运维管理员 | 登录后台、选择目标主机、执行导入、查询详情、管理服务 |
| `be-supervisor-model` | 鉴权、白名单校验、模板渲染、远端执行、主数据落库 |
| MySQL | 存储用户、登录令牌、登录审计、Supervisor 主数据和导入暂存 |
| `local` 执行器 | 本机读写配置并执行 `supervisorctl` |
| `ansible` 执行器 | 对远端主机执行受控命令和文件操作 |
| 被管理主机 | 提供 `/etc/supervisord.d` 和 Supervisor 运行环境 |

## 3. 认证与令牌流程

认证接口统一挂在 `/admin/api/auth/*`：

- `POST /login`
- `GET /profile`
- `POST /logout`

登录链路：

1. 校验用户名和密码
2. 签发 JWT
3. 写入 `sys_login_token`
4. 更新 `sys_user` 最近登录信息
5. 写入 `sys_login_log`

所有 `/admin/api/*` 请求都要求 Bearer JWT；跨源 `OPTIONS` 预检直接放行，不做鉴权。

## 4. 主机与执行器模型

- 目标主机必须来自 `config.yaml` 的 `hosts`
- 每台主机固定绑定 `local` 或 `ansible`
- `ansible` 主机统一使用 `host.ip` 作为 inventory pattern
- 控制机 `inventoryPath` 必须存在且可读

路径与安全边界：

- 受控目录固定为 `/etc/supervisord.d`
- 只允许操作 `.ini`、`.ini.bak`、`.ini.bak.*`
- 拒绝路径穿越、绝对路径注入和非法文件名

## 5. 主数据与现场边界

数据库主表为 `sys_supervisor_service`，真实定位字段固定为：

- `config_path`
- `file_name`
- `content_program_name`

其他结构化字段包括：

- `job_name/module_name/java_path/active_profile/port/jar_name/xms/xmx/run_user`
- `manage_mode`
- `baseline_content`
- `status/pid/uptime`
- `command/directory/stdout_logfile`
- `config_content/backup_config_content`
- `is_archived/archived_at/restored_at`

边界规则：

- 列表和详情默认只查数据库
- 远端 `.ini/.bak` 与 `supervisorctl status` 只在显式同步或运行操作中读取
- 概况接口实时采集，但不落库

## 6. 概况、列表、详情与同步

概况：

- `GET /admin/api/supervisor/overview`
- 直接采集目标主机 CPU、内存、基础检查
- 不可达时返回 `200 + UNREACHABLE`
- 前端不允许回退到 mock 数据

服务列表：

- `GET /admin/api/supervisor/services`
- 纯数据库分页查询
- 默认只看未归档记录
- 排序固定为 `update_time DESC, id DESC`

服务详情：

- `GET /admin/api/supervisor/services/{contentProgramName}`
- 只返回数据库快照

单服务同步：

- `POST /admin/api/supervisor/services/{contentProgramName}/sync`
- 显式读取远端状态、当前配置和可选备份
- 同步结果回写数据库

## 7. 初始化导入流程

导入采用两阶段协议：

1. `GET /imports/staging?host=...`
   - 恢复当前用户该主机最近一次预检结果
2. `POST /imports` with `mode=PRECHECK`
   - 扫描远端 `*.ini`
   - 解析并写入暂存表
   - 返回 `batchId + summary + items`
3. `POST /imports` with `mode=COMMIT`
   - 按 `batchId` 原子提交正式表
   - 成功后删除暂存

规则：

- 暂存按“当前用户 + 主机 IP”隔离
- `PRECHECK` 会覆盖该用户该主机上一批暂存
- `COMMIT` 成功才删暂存
- `COMMIT` 失败时正式表回滚，暂存保留
- 导入只写数据库，不改远端现场

## 8. 新增、修改、归档、删除流程

新增：

1. 校验主机白名单与字段安全
2. 渲染模板
3. 校验数据库冲突、文件冲突和端口冲突
4. 写入 `.ini`
5. 执行 `reread/update`
6. 成功后写主表

修改：

- 允许 `local + ansible`
- 未归档记录才能修改
- 导入记录修改后转成 `TEMPLATE_MANAGED`

归档：

1. `stop`
2. 备份 `.ini -> .bak`
3. 删除 `.ini`
4. `reread/update`
5. 标记 `is_archived=true`

还原：

1. 恢复 `.bak`
2. `reread/update`
3. 不自动启动

硬删除：

- 只允许删除已归档记录
- 先删数据库记录
- 再清理远端 `.ini/.bak`
- 远端清理失败只返回告警，不回滚主表删除

## 9. 配置与部署

配置范式：

- `config.yaml`：主配置
- `.env.dev/.env.prod`：敏感值

日志规则：

- 默认 stdout
- 配置 `app.logPath` 或 `APP_LOG_PATH` 后启用 `stdout + 文件双写`

容器部署：

- 使用 `Dockerfile + docker-compose.yml`
- compose 启动前自动执行 `aerich upgrade`
- 需要宿主机挂载 `/etc/ansible` 和日志目录

## 10. 验证方式

- `.venv/bin/python -m pytest -q`
- `docker compose config`
